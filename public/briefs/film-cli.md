# film-cli

## One-Liner

A quality-gated CLI harness that enables AI agents to autonomously produce multi-shot narrative films.

---

## The Problem

AI video generation models crossed a remarkable quality threshold in 2025. Kling v3-omni can produce photorealistic 5-15 second clips with lip-synced dialogue, ambient sound design, and cinematic camera movement. Runway Gen-3, Google Veo, and OpenAI Sora deliver similar single-clip fidelity. Yet nobody has solved the actual hard problem: producing a coherent 2-minute narrative film from these clips. A narrative film is not a collection of pretty clips. It requires 10-30 shots that share consistent characters (same face, same wardrobe, same skin tone across every angle), continuous story logic (cause and effect that a first-time viewer can follow without explanation), matched visual style (lighting, color grade, lens characteristics), dialogue with accurate lip-sync, and professional assembly with motivated transitions. The gap between "one impressive clip" and "one coherent film" is where every existing tool fails.

The competitive landscape confirms this gap. A deep analysis of eight competing projects (OpenMontage, ViMax, MovieAgent, FilmAgent, MultiShotMaster, and others) revealed a universal architectural flaw: they all implement generate-and-concatenate pipelines. They produce all shots from a script, stitch them together, and call it done. None implements a closed-loop quality gate where a reviewer scores each artifact and feeds actionable prompt-engineering suggestions back into the generation loop. None enforces that Shot N+1 must be designed after seeing Shot N's actual output, not just its intended output. None encodes the 30+ production rules that separate a working AI film from a broken one (such as: "static camera" in a Kling prompt freezes the entire frame, or voice_list entries use a completely different syntax than what the documentation implies). These tools treat film production as a batch process. Film production is a sequential, adaptive, quality-obsessed process. The batch approach produces visual noise that no audience would watch to the end.

The builder's prior work proved the workflow is viable. Two narrative micro-films (22 shots and 13 shots respectively) were shipped using Claude Code driving ad-hoc Python scripts. But the process was fragile in specific, enumerable ways. State lived in conversation context, so when the context window compacted, the agent forgot which shots were done, which versions had failed, and why. Prompts, reviews, and decisions were lost when the conversation ended, making it impossible to learn across sessions. Thirty production rules existed only as natural-language text in the system prompt, meaning nothing prevented the agent from repeating known, expensive mistakes. No quality gates were enforced at the system level, so the agent could skip review, use the wrong model, or forget to pass character reference images. Assembly was manual. film-cli exists to solve every one of these problems.

---

## The Solution

film-cli is a TypeScript/Bun command-line tool that wraps the entire AI filmmaking production pipeline into atomic, quality-gated commands designed for consumption by AI coding agents like Claude Code, Cursor, and Codex. Every command accepts a `--json` flag for structured agent consumption and returns machine-parseable output including the next recommended action. The core insight is architectural: film-cli treats the AI agent as the director and itself as the production infrastructure. The agent makes creative decisions (what story to tell, how to frame each shot, what the dialogue should be). The CLI enforces production discipline (validating prompts against 35 rules before making expensive API calls, requiring review scores above 8/10 before advancing, persisting every prompt/review/decision in SQLite for cross-session continuity, and blocking out-of-order shot production).

The system implements a greedy sequential algorithm that is the single most important design decision in the architecture. Shot N+1 cannot begin until Shot N is accepted. This is not a workflow preference; it is a hard constraint enforced in `src/commands/shot.ts` via the `generateFrame` function, which checks that the previous shot's status is "accepted" before allowing frame generation to proceed. The reason is continuity: in a narrative film, Shot 2's framing, character position, and emotional state depend on what Shot 1 actually produced, not what Shot 1 was intended to produce. AI video models are stochastic. The character might turn left instead of right. The lighting might come out warmer than specified. The greedy algorithm forces the agent to adapt to reality rather than planning against assumptions. This is the fundamental difference between film-cli and every generate-and-concatenate competitor.

The quality gate system uses Gemini 3.1 Pro (via OpenRouter) as an automated film reviewer. Every frame and every video must pass review with a score of 8/10 or higher before advancing. The reviewer receives not just the artifact but also the prompts used to generate it, enabling prompt-engineering feedback rather than vague quality complaints. When a frame scores below threshold, the reviewer's `PROMPT_FIX` field contains specific rewrite suggestions. When a video fails, the reviewer can recommend "reroll" (regenerate with better prompts) or "restructure" (fundamentally redesign the shot). After 6 failed iterations, a shot enters "structural_failure" status, signaling that the creative approach itself needs rethinking. This closed feedback loop between generation, review, and prompt revision is what makes autonomous production possible.

## Architecture Overview

The production pipeline flows from creative brief to rendered film through a series of distinct phases, each managed by dedicated CLI commands. The pipeline begins with project initialization (`film init`), which creates a SQLite database (`film.db`), a TOML configuration file (`film.toml`), and a directory structure for assets, episodes, and knowledge. The creative brief (`film brief set`) and visual DNA (`film brief set-dna`) establish the project's narrative compass and visual language. These are stored in the `projects.config_json` field and injected into every prompt composition step, ensuring visual consistency across all shots without requiring the agent to remember style details across context windows.

Asset registration (`film asset register`) is a prerequisite phase that the system treats as non-optional. Character portraits, turnaround sheets (4-angle reference images), expression sheets, scene establishing shots, and voice clones must be registered before any shot production begins. The `src/commands/asset.ts` module also integrates with Kling's Element Library API (`film asset element-create`) for 3D identity anchoring, which uploads multi-angle reference images to build a persistent character identity that Kling can reference across generations. Every shot's `image_list` must include character portrait references beyond the first frame, enforced by rule R011 in `src/rules.ts`. Without this, every shot generates a different-looking person. This was identified as the number one quality issue across 35+ production shots and is now a hard-coded system constraint.

The core production loop operates per-shot through a state machine defined in `src/state.ts`. Each shot progresses through eight states: `planned`, `frame_generating`, `frame_generated`, `frame_reviewed`, `video_generating`, `video_generated`, `video_reviewed`, and `accepted`. The state machine enforces valid transitions (a shot cannot jump from `planned` to `video_generating`, for instance) and provides a `getNextAction()` function that recommends the next step based on current state, review score, and reroll count. The agent calls `film shot next` to discover which shot needs attention and what action to take. This eliminates the need for the agent to track production state in its own context, making the system resilient to context compaction.

Each shot's production involves four API-intensive steps: frame generation via Nano Banana 2 (Gemini 3.1 Flash Image, accessed through OpenRouter), frame review via Gemini 3.1 Pro, video generation via Kling v3-omni, and video review via Gemini 3.1 Pro. Before each generation step, the rules engine in `src/rules.ts` validates the prompt and API parameters against production rules. Before each review step, the system packages the artifact alongside its generating prompts for context-rich evaluation. The `src/spec.ts` module adds an additional layer: structured shot specifications (YAML or JSON) that decompose every shot into subject, action beats, camera phases, lighting, audio layers, performance direction, and transition planning. The CLI validates spec completeness and rejects vague or incomplete specifications before allowing prompt composition.

Final assembly is handled by `src/commands/assemble.ts`, which offers two engines. The FFmpeg rough-cut engine (`film assemble rough-cut`) concatenates accepted shot videos with automatic audio normalization (a +4dB pre-boost followed by loudnorm targeting I=-12, LRA=7, TP=-1, because Kling generates quiet audio) and re-encoding for platform compatibility (H264 High Profile, yuv420p, faststart). The Remotion engine (`film assemble remotion`) generates a complete React project with cross-dissolve/fade transitions via `@remotion/transitions`, per-speaker subtitle styling with fade-in/fade-out animation, and configurable shot ordering. After rendering, the assembled film can be reviewed by Gemini Pro via `film assemble review` (narrative/pacing/emotion analysis) and `film assemble review-final` (cinema-ready QC that outputs specific FFmpeg and Remotion fix commands with exact timestamps).

## Technical Deep Dive

### The Rules Engine (`src/rules.ts`)

The rules engine is the system's immune system. It contains three validation functions that run automatically before any expensive API call: `validateKlingParams()` for API parameter validation, `validateKlingPrompt()` for video prompt quality, and `validateFramePrompt()` for first-frame prompt quality. Each function returns an array of `RuleResult` objects with severity levels (error, warning, info), and errors block generation entirely. Every rule in the engine corresponds to a specific production failure that occurred during actual film production. Rule R010 ("no_freeze_words") exists because including phrases like "static camera" or "locked camera" in a Kling prompt causes the model to freeze the entire frame, producing a PowerPoint-like output. Rule R007 ("no_voice_token") exists because using `<<<voice_N>>>` tokens (suggested by some documentation) causes Kling to ignore the voice_id entirely, producing the wrong voice or silence. Rule R005 ("three_layer_motion") checks that every Kling prompt contains all three motion layers (camera movement, subject action, atmospheric particles), because missing any one of them produces flat, lifeless output.

The rules engine also includes newer rules derived from analysis of 10 production-tested exemplar prompts stored in `references/exemplar-prompts/`. Rule R020 ("kinetic_verbs") checks that prompts use aggressive physical verbs like "rockets," "slams," "snaps," "whips" rather than passive descriptions like "slowly moves" or "gently turns," because Kling responds dramatically better to high-energy language. Rule R021 ("sfx_narrative") ensures sound effects are timed narrative beats ("door slam at 0:04") rather than ambient wallpaper ("low drone"), because narrative SFX creates tension and punctuation while ambient SFX produces forgettable background noise. Rule R022 ("camera_rig_style") checks for a camera rig specification ("handheld shaky" versus "robotic snappy" versus "steadicam smooth"), because the rig defines how the camera movement feels, not just what it does. Rule R024 ("multi_phase_camera") checks for multi-phase camera movements separated by "then" or "followed by," because exemplar prompts average 2-4 camera phases per shot and single-phase movements produce flat cinematography.

### The Shot Specification System (`src/spec.ts`)

The shot specification system is what the codebase comments call "the anti-laziness enforcer." It addresses a specific failure mode: AI agents generating vague, underspecified prompts that waste API credits on mediocre output. The `ShotSpec` interface defines every element a shot must specify before generation: subject (who is in frame), action (decomposed into visual beats, not abstract descriptions), camera (framing, angle, movement, phases, rig style, focus pulls, lens choice), lighting, a 7-layer audio structure (room tone, foley, SFX, walla, ambience, music direction), performance direction (facial micro-actions and body language, not emotions), voice acting direction (emotion arc, physical tells, delivery style), stability anchors (what the AI must protect from degradation), avoid lists, transition planning, VFX complexity flags, and color grade overrides.

The system enforces quality through the `validateSpec()` function, which runs 20+ checks. It rejects missing required fields. It checks action quality by counting visual-beat clauses (requiring at least 2), checking for body-part specifics (hand, eye, jaw, shoulder), checking for spatial direction (left, right, toward, up, down), and flagging actions shorter than 50 characters. It blocks "static/locked camera" in the camera.move field. It warns when camera phases are missing or fewer than 2. It requires SFX narrative moments (timed sound events with narrative purpose). It requires voice acting direction for dialogue shots. It requires stability anchors. It validates dialogue format (voice_id, Chinese text for Kling, English for subtitles, max 2 voices per shot due to API limits). It checks transition completeness (match cuts need a match_element, J-cuts and L-cuts need audio_overlap_frames). Once validated, the `composeFramePrompt()` and `composeKlingPrompt()` functions automatically generate the NB2 and Kling prompts from the spec, and the `validatePromptAgainstSpec()` function cross-checks that the generated prompt actually covers what the spec defined, catching forgotten elements like camera movement, dialogue, audio, or lighting.

### Quality Gates (`src/integrations/gemini.ts`)

The quality gate system uses Gemini 3.1 Pro as an automated film reviewer. The choice of Pro over Flash is itself a production rule (R020): Flash was tested and found to miss narrative and composition issues, checking only technical attributes like resolution and brightness. Pro evaluates seven dimensions for frame review (narrative intent, character consistency, composition, lighting match, frozen mid-motion quality, AI artifacts, animation suitability) and seven dimensions for video review (narrative clarity, performance quality, camera smoothness, physics accuracy, audio correctness, character consistency, three-layer motion presence). The critical architectural decision is prompt-inclusive review: the reviewer always receives the prompts used to generate the artifact alongside the artifact itself. This enables the reviewer's `PROMPT_FIX` output to contain specific, actionable prompt rewrites rather than vague quality complaints. For video review, the reviewer sees both the NB2 frame prompt and the Kling video prompt, giving it full context for diagnosing whether a failure originated in the frame stage or the video stage.

The assembly review system (`reviewAssembly` and `reviewAssemblyFinal`) evaluates the complete film against nine criteria: story coherence, pacing, transition motivation, audio continuity, character consistency, emotional arc, ending satisfaction, music/score recommendations, and audience engagement. The final QC review (`reviewAssemblyFinal`) operates at broadcast standard and outputs specific, executable fix commands. It analyzes every cut point for flash frames, color temperature mismatches, and spatial geography breaks. It checks audio for loudness compliance (-14 LUFS streaming, -24 LUFS broadcast), volume jumps, pops, and room tone continuity. It flags AI artifacts with exact timestamps and recommends whether each defect is fixable in post-production or requires shot regeneration. It provides FFmpeg commands for color correction, audio ducking, and trim operations, and Remotion code for transition changes and subtitle styling.

### The State Machine (`src/state.ts`)

The state machine defines 10 possible shot statuses and enforces valid transitions between them. The transition map is explicit: `planned` can only transition to `frame_generating`; `frame_reviewed` can go to either `frame_generating` (if the frame failed review) or `video_generating` (if it passed); `video_reviewed` can go to `accepted`, `rerolled`, or `restructured`. The `getNextAction()` function encodes the decision logic: if a frame review score is below 8, regenerate the frame; if a video review score is 8 or above, accept; if the score is 7+ and fewer than 3 rerolls have occurred, reroll; if 6 iterations have been exhausted (controlled by `MAX_ITERATIONS = 6`), declare structural failure. The distinction between "reroll" and "restructure" is meaningful: reroll means trying again with better prompts for the same shot concept, while restructure means fundamentally redesigning the shot (different framing, different action, different approach to the scene). Both reset the shot to `planned` status and create a new version entry, but they signal different agent behaviors.

### Prompt Composition and the Greedy Algorithm

The greedy algorithm is enforced at the code level in `src/commands/shot.ts`. When `film shot generate-frame` is called for shot N where N > 1, the function queries the database for the previous shot's status. If the previous shot is not `accepted`, the command exits with a `GREEDY BLOCK` error. The agent can override with `--force` for legitimate cases like insert shots or parallel scenes, but the default behavior enforces sequential production. The `film shot next` command supports this by returning the next non-accepted shot along with the previous accepted shot's video path, enabling the agent to reference the actual output when designing the next shot.

The `continues_from` field in `ShotSpec` enables frame-level continuity between shots. When shot 5 specifies `continues_from: 4`, the `generateFrame` function extracts the last frame from shot 4's accepted video using FFmpeg (`ffmpeg -sseof -0.1 -i ... -frames:v 1`) and uses it as shot 5's first frame, bypassing NB2 generation entirely. This is critical for scenes that span multiple shots (Kling's maximum duration is 15 seconds) or for reaction shots that must maintain spatial continuity with the preceding action shot. The extracted frame is stored as `continuity_frame.png` in the shot's directory and recorded in the versions table with a `[CONTINUITY]` prefix.

### Image Quality Assessment (`src/integrations/iqa.ts`)

Beyond the subjective Gemini Pro review, the system includes an objective image quality assessment module that uses the Python `pyiqa` library to compute three deterministic metrics: NIQE (naturalness, where lower is better and real photos score 3-4 while CG renders score 5+), MUSIQ (technical quality, where higher is better and good frames score 45+), and NIMA (aesthetics on a 1-10 scale). These are fixed algorithms that produce the same score for the same image every time, providing a quantifiable complement to the LLM-based review. The system classifies images as "photorealistic" (NIQE below 4.0 and MUSIQ above 40), "acceptable," "cg_warning" (looks like a game render), or "poor." The CG warning specifically recommends adding film grain, wet surfaces, and lens imperfections to push the output toward photorealism.

### Remotion Assembly (`src/commands/assemble.ts`)

The Remotion integration generates a complete, self-contained React project that can be previewed in a browser and rendered to a final video. The generated project includes a `package.json` with Remotion 4.x dependencies, a `Root.tsx` that registers the composition at 24 FPS with 1284x716 resolution, and a `FinalCut.tsx` that implements the edit. Each accepted shot becomes a `TransitionSeries.Sequence` with its actual duration (probed via FFmpeg), and transitions between sequences use `@remotion/transitions` with configurable timing. The subtitle overlay system renders active subtitles with animated opacity (4-frame fade-in/out), semi-transparent black background, and system-ui font styling. Both the transition configuration and subtitle data can be supplied as external JSON files, allowing the agent to design the edit independently of the shot production phase. Post-rendering, the output is re-encoded through FFmpeg with H264 High Profile, yuv420p pixel format, CRF 18 quality, and faststart moov atom placement for streaming compatibility.

### Dual Output Mode and Agent Communication (`src/output.ts`, `src/cli.ts`)

Every command in film-cli supports dual output: human-readable text for manual use and structured JSON for agent consumption (activated by the `--json` global flag). The `emit()`, `success()`, `error()`, and `table()` functions in `src/output.ts` handle this branching. When an agent calls `film shot next 1 --json`, it receives a JSON object containing the shot number, current status, recommended next action, latest review score, latest fix suggestion, and the file path to the previous accepted shot's video. This design means the agent never needs to parse human-readable output or maintain its own state tracking. The CLI is the single source of truth for production state, and the agent treats each `--json` response as a self-contained instruction packet.

## Challenges & Solutions

### 1. Character Consistency Across Shots

**Problem:** Without explicit character reference images, AI video models generate a different-looking person in every shot. Face structure, age, skin tone, hair style, and wardrobe drift randomly. This was identified as the number one quality issue across 35+ production shots, and it makes multi-shot narrative completely unwatchable.

**Solution:** film-cli treats character references as a hard requirement, not a nice-to-have. The asset system (`src/commands/asset.ts`) requires registering portraits, turnaround sheets (4-angle references), and expression sheets before production begins. Rule R011 in `src/rules.ts` warns when `image_list` contains only a first frame without character refs. The Kling Element Library integration (`createElement` in `src/integrations/kling.ts`) goes further, uploading multi-angle reference images to build a persistent 3D identity anchor (element_id) that Kling references during generation. Elements are namespaced per project and stored in both the Kling account and the local SQLite database. Every `generateOmniVideo` call includes character refs in the `image_list` and element IDs in the `element_list`.

### 2. The "Frozen Frame" Problem

**Problem:** Kling's video model interprets certain prompt phrases literally in destructive ways. "Static camera," "locked camera," "zero movement," and "nearly motionless" cause the model to freeze the entire output, producing a still image rather than a video. Subtle motion descriptions like "dust settling" or "eyes shifting" are also treated as frozen texture rather than animation. These failures waste API credits and production time.

**Solution:** The rules engine implements layered defenses. Rule R010 (`no_freeze_words`) blocks explicit freeze-trap phrases. Rule R005 (`three_layer_motion`) requires all three motion layers (camera, subject, atmosphere) in every prompt. The `validateSpec()` function in `src/spec.ts` blocks "static/locked camera" in the `camera.move` field with an error-severity result. Rule R020 (`kinetic_verbs`) encourages high-energy verbs because Kling responds better to "pumps arms aggressively" than "moves slowly." The failure modes database in `src/commands/rules.ts` documents the exact symptom ("entire frame frozen, PPT-like output") and fix ("remove locked/static camera, always specify movement") so the agent can learn from past failures.

### 3. Context Loss Across Agent Sessions

**Problem:** When an AI agent's context window compacts or a conversation ends, the agent loses track of which shots are complete, which versions failed, what review feedback said, and what production decisions were made. Starting a new session required re-explaining the entire production state.

**Solution:** SQLite persistence eliminates this problem entirely. Every prompt, review score, review text, one-fix suggestion, prompt-fix suggestion, decision (accept/reroll/restructure), and decision reason is written to the database immediately. The `film status` command provides a complete project dashboard. The `film shot next` command returns the exact next action with full context. The `film shot status` command shows a shot's complete version history with all review scores. An agent starting a fresh session can call `film status --json` and `film shot next 1 --json` and have complete operational context within two commands. No tribal knowledge required.

### 4. Prompt Quality Degradation

**Problem:** AI agents generating prompts for AI video models tend to produce vague, underspecified, or technically incorrect prompts that waste API credits. Common failures include: forgetting camera movement (producing static output), using abstract emotional descriptions instead of visible micro-actions ("sad expression" produces nothing, "eyes close briefly, jaw tightens" produces acting), starting frame prompts with title text (NB2 renders these as on-screen watermarks), and using timestamp sequences in Kling prompts (Kling flattens "0-2s X, 2-3s Y" into one averaged action).

**Solution:** The shot specification system (`src/spec.ts`) forces decomposition before generation. The spec requires action to be broken into visual beats with body-part specifics and spatial directions. The rules engine validates finished prompts against 24+ rules. The `validatePromptAgainstSpec()` function cross-checks that the composed prompt covers every element the spec defined (camera movement, dialogue, audio, lighting, performance direction, film texture words). The exemplar prompts in `references/exemplar-prompts/` provide 10 production-tested templates that demonstrate correct prompt structure, and the rules derived from analyzing those exemplars (R020-R024) encode their patterns as machine-checkable constraints.

### 5. Audio Quality and Normalization

**Problem:** Kling generates audio that is consistently too quiet for playback, and audio volume levels jump between concatenated shots. Unwanted background music sometimes appears despite negative prompts. Voice tone can drift between clips generated in separate API calls.

**Solution:** The assembly pipeline applies automatic audio post-processing: a +4dB pre-boost followed by EBU R128 loudness normalization (I=-12, LRA=7, TP=-1) via FFmpeg's `loudnorm` filter. The assembly review system specifically checks for volume jumps between shots with exact timestamp reporting. The failure modes database documents the unwanted BGM issue and recommends acceptance or post-production filtering. Production rule R009 recommends single long takes (up to 15 seconds) instead of splitting dialogue across multiple clips, because voice tone consistency is better within a single generation than across separate API calls.

## What I Learned

The most important lesson is that AI filmmaking is 20% generation and 80% production infrastructure. The video models are powerful but stochastic. Without systematic quality gates, persistent state, production rules, and structured specifications, the output is random and the process is chaotic. The CLI is not a wrapper around API calls. It is a production discipline engine that transforms unreliable generative AI into a repeatable filmmaking process.

Prompt engineering for video models is a craft with non-obvious rules. "Static camera" does not mean "camera holds still while subject moves." It means "everything freezes." "Sad expression" does not mean "the character looks sad." It means nothing happens on screen. Timestamps in prompts do not mean "do X then Y." They mean "average X and Y into one muddled action." These rules cannot be discovered by reading documentation. They can only be discovered by burning API credits and observing failures. Encoding them into a machine-checkable rules engine means each failure only needs to happen once.

The greedy sequential algorithm is the most counterintuitive and most important architectural decision. Batch-generating all shots from a script is faster and seems more efficient. But narrative films are not independent clips. Every shot exists in the context of what came before it. When Shot 3 produces unexpected camera movement or lighting, Shot 4 must adapt to maintain continuity. The greedy approach is slower but produces coherent stories. The batch approach is faster but produces visual noise.

Review quality varies dramatically between models. Gemini 3.1 Flash is fast and cheap but evaluates only surface-level technical attributes (is the image sharp? is it the right aspect ratio?). Gemini 3.1 Pro evaluates narrative intent, acting quality, and compositional storytelling. The difference between a 6/10 film and an 8/10 film is not technical quality. It is whether a first-time viewer can follow the story and feel something. Only Pro catches those failures.

The shot specification system solves the "lazy prompt" problem that appears in every AI-agent-driven workflow. Without structured requirements, agents naturally produce the minimum viable prompt. The spec forces decomposition: you cannot specify "camera moves" without specifying phases, rig style, and focus pulls. You cannot specify "the character is sad" without specifying facial micro-actions, body language, and emotional arc. The decomposition itself improves quality because it forces the agent to think through the shot before generating it.

## Motivation & Context

The question that started this project was: "Can an AI agent direct a film?" Not generate a single clip. Not animate a character. Direct a film: plan it, shoot it, review it, iterate on it, and assemble it into something a stranger would watch to the end. The answer is yes, conditionally. The AI agent (Claude Code) brings creative decision-making, prompt writing, and adaptive planning. But it needs production infrastructure to be effective. It needs persistent state so it can resume after context loss. It needs a rules engine so it does not repeat expensive mistakes. It needs quality gates so it does not accept mediocre output under time pressure. It needs structured specifications so its prompts are thorough rather than lazy.

film-cli is that infrastructure. It encodes the hard-won production knowledge from two shipped micro-films into a system that any AI agent can use. The CLI is the interface. SQLite is the memory. The rules engine is the discipline. The quality gates are the editorial standards. Together, they transform an AI agent from a clip generator into a film director.

The micro-drama market is estimated at $11 billion and growing rapidly. Current production requires human experts managing dozens of API calls across multiple services. film-cli demonstrates that the production pipeline can be systematized and automated, opening this market to AI-driven content creation at scale.

## Status

film-cli is a working system that has been validated through production use. The complete pipeline from initialization through shot-by-shot production to final assembly with Remotion is implemented and functional. The codebase consists of approximately 3,500 lines of TypeScript across 18 source files organized into CLI commands (`src/commands/`), API integrations (`src/integrations/`), and core modules (`src/rules.ts`, `src/spec.ts`, `src/state.ts`, `src/db.ts`, `src/output.ts`). The rules engine encodes 35 production rules derived from real production failures. The exemplar prompts library contains 10 production-tested templates. The system integrates three external APIs (Kling v3-omni for video, Gemini 3.1 Flash Image for frames, Gemini 3.1 Pro for review) through a unified configuration system (`film.toml`). The SQLite schema tracks projects, characters, assets, episodes, shots, versions, reviews, decisions, and assemblies with full relational integrity and index coverage. The Remotion assembly generates a standalone React project with transitions and subtitles. The FFmpeg pipeline handles audio normalization, video re-encoding, and frame extraction. All commands support dual output (human-readable and JSON) for both manual and agent-driven usage.
